# 1. 摘要
本请求意见稿（RFC）明确了 PitWall Agent 项目的完整技术栈选型。

选型目标是搭建一套现代化、可落地生产的架构，支撑以下能力：
- AI 智能体编排调度
- 检索增强生成（RAG）
- 外部工具调用
- 长期可维护性
- 容器化部署
- 持续集成
- 未来可扩展性

技术选型的评估维度不仅包含功能特性，还涵盖生态成熟度、社区接受度、生产就绪程度以及长期维护成本。

---

# 2. 设计目标
入选的技术栈需满足以下目标。

## G1 生产就绪
所有入选框架都需已在生产环境中被广泛应用。
除非具备压倒性优势，否则不选用实验性框架。

---

## G2 原生 AI 设计
技术栈优先适配 AI 应用开发，而非传统增删改查（CRUD）系统。
架构需原生支持以下能力：
- 大语言模型（LLM）
- 智能体（Agent）
- 检索增强生成（RAG）
- 工具调用
- 流式传输
- 记忆管理

---

## G3 模块化
所有组件都支持独立替换。
示例：将向量数据库从 PostgreSQL + pgvector 更换为 PGVector 时，只需最少的代码改动。

---

## G4 开源友好
项目应避免过度的厂商绑定。
开发者可在本地完整部署整套项目。

---

## G5 行业适配性
技术栈需符合当前行业最佳实践。
每个组件的选型都应有清晰的架构层面依据。

### 技术栈选型中功能特性与开发效率的平衡思路
技术栈选型的核心平衡逻辑，是在**长期架构价值**与**短期交付速度**之间，找到适配项目阶段、团队规模与核心诉求的平衡点，结合本项目的选型逻辑，可从5个维度落地：

1.  **锚定核心诉求，区分「必需功能」与「增值特性」**
    先明确项目的不可替代能力边界：本项目是 AI Agent 系统，Agent 编排、工具调用、流式输出、状态持久化属于核心刚需，必须优先保证功能完备性，选择能力匹配度最高的组件；而可视化配置、后台管理等非核心能力，可通过轻量方案实现甚至暂时省略，换取开发效率。
    核心功能不盲目追求“最强”，但必须成熟稳定；非核心功能不提前过度设计，避免为低频特性增加开发负担。

2.  **优先选择高生态复用度的技术，减少重复造轮子**
    功能的实现尽量依托成熟生态，而非从零自研。比如选择 LangGraph 虽然学习曲线更陡，但它原生自带状态管理、检查点、重试、流式输出、全链路监控等全套能力，无需自行实现工作流引擎，长期来看反而大幅提升开发效率；如果选择轻量简易的框架，后续补全这些生产级能力会消耗更多成本。
    同时统一技术栈生态（如全 Python 后端、OpenAI 兼容协议），减少跨语言、跨协议的适配成本，用统一工具链提升日常开发效率。

3.  **通过松耦合+抽象层，兼顾扩展性与开发效率**
    对易变组件做抽象层封装：比如 LLM 层采用 OpenAI 兼容协议、向量数据库层统一检索接口，既保留了功能上的可替换性（未来切换模型、数据库无需重构核心逻辑），又不会在开发初期过度设计，当下仅需实现一套适配逻辑。
    模块化拆分核心能力，优先落地主链路，非核心模块渐进式增强，避免一开始追求全量功能导致开发周期拉长。

4.  **匹配团队规模与技术能力，不盲目追求技术先进性**
    本项目为单人开发，因此刻意规避了运维复杂度高、学习成本过大的技术方案：比如用 Docker Compose 替代 Kubernetes，用 uv 替代复杂的依赖管理体系，都是用更低的运维/学习成本，换取更高的日常开发效率。
    团队熟悉的技术，即使不是“功能天花板”，只要能覆盖核心需求，也可优先选择——开发效率的提升往往来自技术熟练度，而非技术本身的功能上限。

5.  **分阶段调整权重，用迭代替代一步到位**
    - 原型验证期：开发效率优先，可简化非核心功能，快速跑通核心链路。比如 RAG 初期可先用基础检索，先验证问答流程，再逐步补全重排、混合检索等高级特性。
    - 生产迭代期：逐步补全功能特性、优化架构。比如数据量增长后，再从轻量向量库切换到 PostgreSQL + pgvector，而非初期就上重型组件。
    核心原则：核心能力不妥协，非核心能力可渐进增强，用迭代节奏平衡两者。

# 8. 智能体框架
## 问题背景
PitWall Agent 本质上是一套 AI 智能体系统，而非传统聊天机器人。
所选框架必须支持以下能力：
- 复杂工作流编排
- 工具调用
- 条件路由
- 多步推理
- 记忆管理
- 人机回环（未来规划）
- 流式输出
- 中断/恢复执行
- 状态持久化

同时，框架需支撑项目未来向多智能体架构演进，保持可维护性。

## 候选方案
- LangChain AgentExecutor
- LangGraph
- CrewAI
- AutoGen
- PydanticAI
- Semantic Kernel

## 方案1 — LangChain AgentExecutor
### 优势
- 上手简单
- 生态资源丰富
- 社区体量庞大
- 适合快速搭建原型

### 劣势
- 工作流自定义能力有限
- 执行过程可视化难度大
- 对复杂状态流转支持薄弱
- 并非面向生产级智能体编排设计

### 结论
适用于演示场景和简单助手类项目，不适合本项目。

---

## 方案2 — LangGraph
### 优势
- 基于图的显式工作流建模
- 有状态执行机制
- 支持条件路由
- 原生支持重试能力
- 支持检查点持久化
- 原生支持流式输出
- 原生适配人机回环场景
- 与 LangSmith 集成度极高

### 劣势
- 学习曲线更陡峭
- 相比 AgentExecutor 需要更多样板代码

### 结论
专为生产级 AI 智能体场景设计。

---

## 方案3 — CrewAI
### 优势
- 多智能体搭建门槛极低
- 基于角色的协作模式
- 代码可读性好

### 劣势
- 工作流灵活性有限
- 生态成熟度偏低
- 深度定制难度大

### 结论
适合演示场景，对于高度定制化的生产系统适配性不足。

---

## 方案4 — AutoGen
### 优势
- 多智能体对话能力突出
- 微软技术背书
- 通信模式灵活

### 劣势
- 架构以对话交互为核心
- 对确定性工作流适配性较差
- 调试难度更高

### 结论
更适合科研实验场景。

---

## 方案5 — PydanticAI
### 优势
- 类型安全性出色
- API 设计现代化
- 与 Pydantic 深度集成

### 劣势
- 生态仍在发展阶段
- 工作流编排能力弱于 LangGraph

### 结论
技术前景广阔，但对于本项目而言成熟度不足。

---

## 方案6 — Semantic Kernel
### 优势
- 面向企业级场景设计
- 适配微软技术生态
- 支持规划器能力

### 劣势
- Python 社区体量较小
- 更适配 .NET 技术环境

---

## 选型结论
LangGraph

## 选型理由
LangGraph 在灵活性、可观测性、生态成熟度与生产就绪度之间实现了最佳平衡。
它基于图的执行模型，可以自然地表达复杂工作流，例如：
用户提问
↓
意图识别
↓
任务规划
↓
工具选择
↓
工具执行
↓
逻辑推理
↓
响应输出
不会将执行逻辑隐藏在不透明的智能体循环中。

## 权衡取舍
选择 LangGraph 会提升初始实现复杂度。
但显式的工作流设计能显著提升系统的可维护性、调试效率、测试性与可观测性，这些长期收益远大于额外的开发投入。

## 未来演进
未来版本可在不重构整体架构的前提下，新增以下能力：
- 多智能体管控节点
- 分布式智能体
- 人工审批节点
- 并行工具执行

# 9. 大语言模型服务商
## 问题背景
PitWall Agent 不应与单一 LLM 服务商深度绑定。
开发者应可根据性能、成本、延迟、部署环境自由切换模型。

## 候选方案
- OpenAI
- Anthropic
- Google Gemini
- DeepSeek
- 通义千问（Qwen）
- 本地模型

## 评估标准
服务商必须支持：
- 工具调用
- 流式输出
- 长上下文
- 稳定的 API 服务
- 优先兼容 OpenAI 接口规范

## 选型结论
兼容 OpenAI 协议的 API

## 选型理由
PitWall Agent 不将系统绑定到特定厂商，而是基于 OpenAI API 规范构建抽象层。
这一设计可实现服务商之间的无缝切换，仅需极少的代码改动。

支持的服务商包括：
- OpenAI
- Azure OpenAI
- DeepSeek
- 通义千问
- 硅基流动（SiliconFlow）
- OpenRouter
- Ollama（通过兼容层适配）

## 方案收益
- 无厂商绑定风险
- 模型迁移成本低
- 便于多模型效果对比实验
- 成本优化空间更大

- PostgreSQL + pgvector

- Weaviate

## Chroma

- 轻量易上手

劣势

- 更适合原型验证场景

---

## Pinecone
优势
- 全托管服务
- 高可用性

劣势
- 付费服务
- 厂商绑定
- 依赖公网环境

---

## Weaviate
优势
- 功能特性丰富

劣势
- 运维复杂度更高

---

## PGVector
优势

劣势

## 选型理由
PostgreSQL + pgvector 在扩展性、性能与生产就绪度之间实现了最佳平衡。
如果未来文档量级大幅增长，它也支持向分布式部署架构平滑演进。

## 未来演进
如果文档量级始终保持在较小规模（10万分片以内），未来可推出轻量化部署方案，可选支持 PGVector 以简化基础设施。
