# 1. 项目愿景
一级方程式赛事每个比赛周末都会产生海量信息。

车迷需要在官方网站、新闻媒体、技术规则、计时页面、社交媒体和社区讨论之间反复切换，才能理清赛事进展。

大语言模型虽然能回答 F1 相关的通用问题，但存在三大核心局限：
- 缺乏实时知识储备
- 无法可靠地引用 FIA 官方规则
- 讨论赛事事件或技术规则时容易产生幻觉内容

PitWall 智能助手旨在通过大语言模型推理与权威外部知识源相结合的方式，解决上述问题。

系统不单纯依赖语言模型自身能力，而是从多个可信数据源动态获取信息，让模型基于经过验证的数据完成推理后再生成回答。

最终打造一款 AI 原生的 F1 智能助手，能够输出准确、可溯源、时效性强的答案。

# 2. 项目目标
PitWall 智能助手围绕以下目标设计：

## O1 精准信息检索
系统应尽可能从可信外部源获取事实信息，而非仅依赖大语言模型的内部知识。

典型检索场景包括：
- FIA 赛事运动规则
- FIA 技术规则
- 官方赛事日程
- 车手积分榜
- 车队积分榜
- 官方新闻
- 车队公告

---

## O2 可解释的回答
所有与规则相关的回答都必须附带清晰的引用来源。

系统基于 FIA 官方文件作答时，必须提供：
- 文件名称
- 条款编号
- 页码
- 必要时附上原文节选

确保用户可以独立验证每一条回答的真实性。

---

## O3 智能工具编排
项目采用单智能体架构，而非构建多个独立的 AI 智能体。

智能体会根据用户意图，动态选择并调度各类专用工具。

典型工具包括：
- 新闻检索工具
- 赛事信息工具
- 规则检索工具（RAG）
- 策略分析工具

该架构在降低维护成本的同时，具备优秀的可扩展性。

---

## O4 生产级可用性
本项目并非概念验证演示项目。

它遵循面向生产环境的工程实践，包括：
- 分层架构
- 模块化设计
- 结构化日志
- 容器化部署
- 持续集成
- 自动化测试
- 配置管理
- 可观测性

---

## O5 可扩展性
架构应支持新增能力，且无需修改现有业务逻辑。

未来可扩展能力包括：
- 遥测数据分析
- 语音交互
- 图像理解
- 多模态输入
- 消息推送
- 多语言支持

新增工具时，核心架构应保持稳定。

# 3. 项目范围
PitWall 智能助手聚焦于通过多源知识的智能推理，为 F1 用户提供辅助服务。

项目当前包含以下能力领域：

## 实时新闻
从可信来源检索并汇总最新的 F1 赛事新闻。

典型问题包括：
- 今天围场内发生了什么？
- 法拉利车队本周末在测试什么新部件？
- FIA 是否发布了新的技术指令？

---

## 赛事情报
提供官方赛事相关信息，包括：
- 赛历
- 比赛周末日程
- 各阶段比赛时间
- 车手积分榜
- 车队积分榜
- 正赛成绩
- 冲刺赛成绩

---

## FIA 规则问答
支持用户用自然语言查询 FIA 官方文件。

典型问题包括：
- 什么是封闭检车区（Parc Fermé）？
- 这位车手为什么会被处罚？
- 底板磨损的限制标准是什么？
- 如何界定维修站不安全放行？

系统会先通过检索增强生成（RAG）调取相关文档章节，再生成回答。

---

## 策略分析
结合规则知识与赛事信息，分析战术决策。

典型问题包括：
- 法拉利应该在安全车时段进站吗？
- 迈凯伦为什么选择采用 undercut（早进站超越）战术？
- 虚拟安全车阶段对哪方更有利？

这类回答同时依赖检索到的知识与大语言模型的推理能力。

---

# 4. 核心原则
PitWall 智能助手的架构遵循以下原则：

## AI 原生
人工智能是系统的核心，而非附加功能。

应用围绕智能体运行时设计，具备推理、规划、调度外部工具的能力，而不仅是文本生成。

---

## 先检索、后生成
但凡需要事实信息，系统都会先调取外部知识，再调用语言模型。

大语言模型应基于经过验证的上下文推理，而非依赖自身内部记忆。

该原则能大幅降低幻觉，提升回答的可靠性。

---

## 工具化推理
大语言模型不应直接访问外部资源。

所有外部能力都封装为独立工具。

典型工具包括：
- 新闻工具
- 赛事工具
- 规则工具
- 策略工具

由智能体运行时判断每个请求需要调用哪些工具。

---

## 可解释性
回答应始终可溯源。

当知识来自官方文档时，回答应尽可能附带引用来源。

用户不仅能知道答案，还能清楚答案的出处。

---

## 职责分离
每个子系统都有清晰明确的职责边界。
- 前端专注于用户交互
- 后端对外提供应用服务
- 智能体运行时负责推理
- 工具层负责与外部系统通信
- 基础设施层负责存储与部署

任何组件都不应承担超出其边界的职责。

---

## 可扩展性
新增能力应通过添加新工具或服务实现，而非修改现有组件。

架构优先采用组合式扩展，而非修改式开发。

---

## 智能问答

- 工具输出结果

---

## 引用支持

# 7. 高层架构
PitWall 智能助手采用分层架构。

```
表现层
↓
应用层
↓
智能体运行时
↓
工具层
↓
知识与基础设施层
```

各层职责如下：

## 表现层
负责所有面向用户的交互界面。

典型组件包括：
- Web 前端界面
- 聊天交互界面
- 流式输出
- Markdown 渲染

---

## 应用层
负责业务流程的协调调度。

职责包括：
- API 接口
- 会话管理
- 身份认证
- 请求校验
- 服务编排

应用层与智能体逻辑刻意保持解耦。

---

## 智能体运行时
智能体运行时是系统的推理引擎。

基于 LangGraph 实现，负责：
- 意图分析
- 任务规划
- 工具选择
- 工具编排
- 上下文构建
- 回答生成

智能体运行时从不直接与数据库或外部 API 通信，所有外部交互都通过工具层完成。

---

## 工具层
工具层封装所有外部能力。

典型工具包括：
- 新闻工具
- 赛事工具
- 规则工具
- 策略工具

每个工具都提供稳定的调用接口，与底层实现无关。

---

## 知识与基础设施层
负责持久化存储与知识检索。

组件包括：
- PostgreSQL
- Redis
- PostgreSQL + pgvector
- 对象存储
- 外部 API

该层与业务逻辑完全隔离。

# 10. 工程原则
项目开发全程遵循以下工程原则：

## 模块化设计
每个模块都应具备单一、清晰的职责。
业务逻辑绝不与基础设施组件紧耦合。

---

## 低耦合
组件通过稳定的接口通信。

替换某一项实现时，不应要求修改不相关的模块。

例如：
- 替换嵌入模型
- 更换向量数据库
- 切换大语言模型服务商

都应只需要最小范围的改动。

---

## 高内聚
相关职责应归属同一模块。

例如：
- 所有检索逻辑归属于知识模块
- 所有编排逻辑归属于智能体运行时
- 所有 HTTP 接口归属于 API 模块

---

## 配置优先，避免硬编码
环境相关的配置必须外置。

例如：
- API 密钥
- 数据库连接地址
- 模型名称
- 服务端点

敏感信息不得提交至代码仓库。

---

## 可观测性
系统应提供充足的信息用于调试与监控。

日志、链路追踪、运行指标应作为一等工程需求，而非事后补充。

---

## 可测试性
所有核心组件都应支持独立测试。

项目应支持：
- 单元测试
- 集成测试
- 端到端测试

且无需大量改造代码。
